# V2 提示词优化方案

## 目标

让产品保持“先方案、后完整短篇、自动保存进编辑器”的极简体验。默认生成约 2 万字的完整短篇小说。

## 模型分工

- Kimi：负责用户可见的故事方案和最终正文。正式完整短篇正文只能由 Kimi K2.6 分段生成，不使用其他模型替写。
- DeepSeek：负责后台蓝图、市场结构和连续性检查。DeepSeek 不写正文，也不在 Kimi 失败时救场写正文。
- 本地样稿：只用于方案或链路演示，不能作为正式全文自动保存。

## 提示词原则

- 方案阶段只输出标题、方向、大纲、亮点和市场理由，不展示旧十步流程。
- 大纲必须有具体物件、证据、记录或异常处境，拒绝空泛题名。
- 正文按完整短篇蓝图分段生成，每段只写连续小说正文纯文本，不用 JSON 包正文，减少截断和格式污染。
- 每段都有硬字数范围；偏短时先让 Kimi 从上一句后续写补足，仍不足或连续性不合格时由 Kimi 重写该段。
- DeepSeek 每段后只返回连续性检查 JSON 和故事状态更新，包括已发生事件、已揭露信息、读者已知、未回收伏笔和下一段承接点。
- 质量闸门检查：非小说内容、AI 套话、开头过慢、重复信息、缺少具体材料、结尾不稳、分段不承接前文。

## 资料依据

- Kimi K2.6 官方文档：K2.6 支持 thinking / non-thinking 模式；禁用 thinking 时请求体使用 `thinking: {"type":"disabled"}`，并且官方示例使用 `max_tokens`。完整短篇不应依赖一次性生成，而应走稳定的后台分段队列。
- DeepSeek 官方 JSON Output 文档：JSON 模式除了设置 `response_format`，还需要在提示词中明确写 JSON 并提供期望 JSON 示例，且要合理设置 `max_tokens` 防止截断。
- DeepSeek Chat Completion 文档：`thinking` 可设置为 `enabled` 或 `disabled`，默认启用；后台结构任务适合关闭思考模式来降低等待。

## 下一轮可优化

- 为真实生成结果记录 token 用量和耗时。
- 统计每次续写、重写和连续性检查的触发原因，用数据反推提示词。
- 后续可把 2 万字完整短篇从当前后台任务升级为更细的持久化队列，页面只显示简单进度。
