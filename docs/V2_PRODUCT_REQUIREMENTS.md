# 神笔马良 V2 开发需求文档

## 1. 产品定位

神笔马良 V2 是一个本地优先的市场导向短篇小说生成 Agent。

它不是作者个人表达工具，不以用户个人喜好为核心目标；它的核心目标是生成更可能获得平台和读者数据认可的短篇小说初稿。

用户关心的是结果数据，而不是创作过程。因此产品必须极简、自动化、少解释、少面板，避免把内部推理流程暴露成一堆表格或卡片。

## 2. 核心目标

用户只需要选择一种模式并点击开始：

1. 有灵感：用户输入一个灵感、设定、爽点或模糊方向，Agent 自动优化为更适合平台的短篇，并生成全文。
2. 全自动：用户可以什么都不填，也可以给出少量非必选偏好，例如创作方向、字数范围或禁写方向。未填写时，Agent 根据内置市场策略、历史作品反馈、写作记忆和可用数据，自动选择方向并生成全文。

生成完成后，系统必须自动保存作品，并直接跳转到正文编辑器，让用户开始修改正文。

## 3. 非目标

V2 第一阶段不做以下内容：

- 不做数据看板。
- 不做风向标大屏。
- 不做复杂数据源管理主流程。
- 不复用旧的流程框架。
- 不展示过程卡片、结构表格、创作理论术语或中间推理文档。
- 不让用户填写大量参数。
- 不把首页做成驾驶舱。
- 不把产品做成 SaaS、多用户系统、移动端 App、桌面 App 或浏览器插件。
- 不自动登录、抓取或绕过平台后台。
- 不承诺训练 Kimi、DeepSeek 或其他大模型本体。

## 4. 第一原则

### 4.1 市场优先

Agent 生成作品时，不追求用户个人审美表达，而追求平台读者更可能点击、读完、收藏、评论和继续追读的内容。

所有后台判断都应围绕以下问题：

- 开头是否有点击欲。
- 前 300 字是否能留住读者。
- 冲突是否足够强。
- 主角目标是否清晰。
- 情绪是否持续升级。
- 每一场是否推进信息或关系变化。
- 是否存在读者弃文风险。
- 结尾是否有爽点、反转、余味或强情绪落点。

### 4.2 前端极简

用户不需要看到 Agent 怎么工作。前端最多显示当前进度，不展示中间推理文档、结构卡片或分析报告。

用户的主路径必须是：

进入首页 -> 选择模式 -> 点击开始 -> 等待进度 -> 自动进入编辑器。

### 4.3 全自动优先

Agent 应自己完成市场选择、故事承诺设计、正文生成、自检、必要重写和保存。

除非生成失败或用户主动停止，否则中途不要求用户确认。

### 4.4 正文优先

最终交付物是完整可编辑正文，不是创作方案。

后台可以保留必要的调试数据和生成元数据，但前端不把过程文档作为主要结果展示。

## 5. 用户角色

### 5.1 主用户

单人本地使用的短篇小说创作者。

用户希望快速产出市场导向短篇初稿，然后在编辑器里人工修改。

### 5.2 用户心智

用户不关心 Agent 内部用了什么流程，也不关心页面是否有复杂数据模块。

用户只关心：

- 能不能自动写。
- 写得像不像可发布短篇。
- 能不能少操作。
- 能不能直接进入正文编辑。
- 后续能不能根据数据和复盘变得更有效。

## 6. 核心功能范围

### 6.1 首页

首页是唯一主入口。

首页只包含：

- 产品名或极简标识。
- 两个模式按钮：
  - 有灵感
  - 全自动
- 有灵感模式下显示一个输入框。
- 一个主按钮：
  - 开始生成全文
- 一个极简进度区域。

首页不展示：

- 数据图表。
- 作品统计。
- 趋势卡片。
- 参数模板。
- 记忆列表。
- 数据源入口。
- 过程卡片或结构表格。

### 6.2 有灵感模式

用户输入任意短文本，可能是：

- 一个题材。
- 一个冲突。
- 一个角色。
- 一个反转。
- 一句设定。
- 一段乱写的灵感。

Agent 必须自动完成：

1. 判断灵感是否有市场潜力。
2. 修正或强化卖点。
3. 选择更适合平台读者的题材包装。
4. 设计高点击开头。
5. 设计持续冲突。
6. 生成完整正文。
7. 自检并自动重写明显弱段。
8. 保存作品。
9. 跳转编辑器。

用户不需要选择题材、篇幅、文风、主角类型、情绪方向等参数。

### 6.3 全自动模式

用户点击开始后，Agent 默认不要求用户输入方向。

全自动模式可以提供少量非必选选项：

- 创作方向：例如悬疑反转、现实情感、强冲突短剧感、女性成长、男频逆袭等。
- 字数范围：默认自动生成完整短篇，约 2 万字；可选 8000 字、1.5 万字、2 万字。
- 平台倾向：第一阶段默认番茄，可不展示或放在高级设置。
- 禁写方向：例如不要校园、不要古代、不要恐怖、不要出轨等。

这些选项都不能成为必填项。用户不选时，Agent 必须自己决策。

Agent 必须自动完成：

1. 从市场策略库、历史作品反馈、写作记忆中选择一个更可能有效的方向。
2. 生成面向读者的核心故事承诺。
3. 生成主角处境、核心矛盾、关键反转和结尾情绪落点。
4. 生成完整正文。
5. 自检并自动重写明显弱段。
6. 保存作品。
7. 跳转编辑器。

全自动模式必须避免随机感。它应能说明自己选择方向的依据，但该说明默认不展示，只保存到作品元数据里。

### 6.4 生成进度

前端只显示短进度，不展示详细过程。

建议进度文案：

- 正在判断市场方向
- 正在确定高点击卖点
- 正在设计主冲突
- 正在生成正文
- 正在自检弱段
- 正在保存作品
- 正在进入编辑器

进度区域可以显示耗时，但不显示内部结构细节。

### 6.5 编辑器

生成完成后自动进入编辑器。

编辑器保留以下能力：

- 查看完整正文。
- 直接修改正文。
- 标记不满意片段。
- 让 Agent 改写选中片段。
- 应用改写版本。
- 保存版本历史。

编辑器改写只服务正文质量，不展示复杂分析面板。

### 6.6 作品保存

每次生成必须保存为一个作品。

作品至少包含：

- 标题。
- 正文。
- 模式：有灵感 / 全自动。
- 生成时间。
- 使用模型。
- 生成状态。
- 简短市场策略摘要。
- 自检结果摘要。
- 版本历史。

PostgreSQL 可用时写入数据库；不可用时使用本地文件兜底，但 UI 不应把兜底伪装成数据库持久化。

### 6.7 写作记忆

写作记忆保留，但不作为主页面。

系统只沉淀有用信息：

- 哪类开头更可能有效。
- 哪类题材或冲突可复用。
- 哪些写法导致弱节奏。
- 用户人工改稿偏好。
- 作品复盘后的失败原因。
- 下一篇应优先避免的问题。

不沉淀低质量废话，不把每次生成的中间文本都记忆化。

### 6.8 设置

设置页保留，但不是主路径。

设置页只需要支持：

- 选择 AI 供应商。
- 填写 API Key。
- 选择模型。
- 测试连接。
- 查看本地保存状态。

本地诊断、备份、清理等功能可以折叠到高级区域。

## 7. 后台 Agent 流程

前端不展示以下流程，但后端必须执行。

注意：V2 不沿用旧的固定步骤框架。后台流程围绕“市场机会 -> 故事承诺 -> 正文交付 -> 自我质检 -> 保存编辑”重新设计。

### 7.1 市场机会 Agent

职责：

- 判断当前输入或自动方向是否适合短篇平台。
- 从候选方向中选择更可能被读者点击和读完的方向。
- 避免平淡、散、慢热、无冲突、无明确读者欲望的方向。
- 将模糊灵感改造成更清晰的市场机会。

输出：

- 市场方向。
- 核心读者欲望。
- 高点击卖点。
- 主要弃文风险。

### 7.2 故事承诺 Agent

职责：

- 把市场机会转成一个清晰的故事承诺。
- 明确读者点进来后期待看到什么。
- 设计主角处境、压迫来源、反击路径和结尾回报。
- 保证故事不是概念堆砌，而是有明确阅读驱动力。

输出：

- 故事承诺。
- 开头钩子。
- 主冲突。
- 结尾情绪回报。

### 7.3 叙事推进 Agent

职责：

- 组织正文的连续推进。
- 确保每个段落或片段都带来信息、关系、处境或情绪变化。
- 避免纯设定堆砌。
- 避免“开头强、中段散、结尾赶”的常见失败。

输出：

- 隐藏叙事路线。
- 关键推进节点。

### 7.4 正文 Agent

职责：

- 生成完整正文。
- 保持连续叙事。
- 不输出解释性创作说明。
- 不复制已有作品。
- 不在正文里暴露内部计划。

输出：

- 完整正文草稿。

### 7.5 质检 Agent

职责：

- 检查开头、节奏、冲突、人物、情绪、结尾。
- 标记弱段。
- 判断是否需要自动重写。
- 判断正文是否像面向平台读者的小说，而不是像创作方案。

输出：

- 简短自检摘要。
- 需要重写的片段列表。

### 7.6 重写 Agent

职责：

- 自动重写明显弱段。
- 保证重写后正文连贯。
- 不把多版方案暴露给用户，除非用户在编辑器中主动请求。

输出：

- 最终正文。
- 重写记录。

## 8. 模型策略

默认策略：

- Kimi 用于长正文生成。
- DeepSeek 用于市场判断、故事承诺设计、质检或低成本辅助任务。

如果只启用一个模型，也必须能完整跑通。

模型调用失败时：

- 自动重试一次。
- 失败后给出清楚错误。
- 不把本地 mock 当成真实可发布正文。

## 9. 页面与导航调整

### 9.1 第一阶段保留页面

- `/`：V2 首页，唯一主入口。
- `/editor`：正文编辑器。
- `/settings`：折叠式设置。
- `/works`：可选保留为历史作品入口，但不放在主视觉中心。

### 9.2 第一阶段隐藏或移除主导航入口

- `/dashboard`
- `/trends`
- `/sources`
- `/datasources`
- `/memory`
- `/review`
- `/inspiration`
- `/auto` 的复杂参数页

这些能力可以在后台保留，但不作为 V2 的用户主流程。

## 10. API 需求

### 10.1 新增生成接口

建议新增：

```txt
POST /api/generate/full-draft
```

请求：

```json
{
  "mode": "inspiration | autopilot",
  "inspiration": "用户灵感，可为空",
  "targetPlatform": "fanqie",
  "targetLength": "auto | 8000 | 15000 | 20000",
  "optionalDirection": "可选创作方向",
  "avoid": ["可选禁写方向"]
}
```

说明：

- `mode=inspiration` 时 `inspiration` 必填。
- `mode=autopilot` 时 `inspiration` 可为空。
- `targetLength`、`optionalDirection`、`avoid` 都是非必填项；`targetLength=auto` 表示完整短篇，默认约 2 万字。
- 未提供可选项时，Agent 必须基于市场策略自动决策。

响应：

```json
{
  "workId": "string",
  "editorUrl": "/editor?workId=...",
  "title": "string",
  "status": "completed",
  "providerMode": "kimi | deepseek | fallback",
  "marketSummary": "string",
  "qualitySummary": "string"
}
```

### 10.2 进度接口

建议支持：

```txt
GET /api/generate/jobs/:id
```

返回：

```json
{
  "jobId": "string",
  "status": "queued | running | completed | failed",
  "progressLabel": "正在生成正文",
  "workId": "string | null",
  "editorUrl": "string | null",
  "error": "string | null"
}
```

MVP 可先用单请求完成；如果真实生成超过 30 秒，应改成 job + polling。

## 11. 数据结构需求

作品模型需支持以下字段或等价结构：

- `id`
- `title`
- `content`
- `mode`
- `sourceInspiration`
- `targetPlatform`
- `providerMode`
- `modelName`
- `marketSummary`
- `qualitySummary`
- `internalPlan`
- `optionalDirection`
- `targetLength`
- `avoid`
- `createdAt`
- `updatedAt`

其中 `internalPlan` 不在前端主流程展示，仅用于后续复盘、调试和改进。它不是固定步骤结构，不应再承载旧流程。

## 12. 质量标准

一篇生成稿合格的最低标准：

- 有完整标题。
- 有完整正文，而不是大纲。
- 开头 300 字内出现强冲突或强异常。
- 主角目标清晰。
- 每一段剧情有推进。
- 不出现明显模板话术。
- 不出现“以下是”“我将为你”等 AI 说明。
- 不把内部计划、分析标题或结构术语混入正文。
- 结尾有情绪落点。
- 自动保存并跳转编辑器。

## 13. 验收标准

### 13.1 有灵感模式验收

输入：

```txt
女主发现丈夫把她的病历卖给了保险公司
```

期望：

- 页面只显示进度。
- 完成后自动跳转编辑器。
- 编辑器中出现完整正文。
- 正文开头有明确钩子。
- 不展示内部流程。
- 作品被保存。

### 13.2 全自动模式验收

用户不输入任何灵感，只点击开始。

期望：

- Agent 自动选择市场方向。
- 页面只显示进度。
- 完成后自动跳转编辑器。
- 编辑器中出现完整正文。
- 作品元数据保存市场策略摘要。

### 13.3 全自动可选项验收

用户只选择：

```txt
创作方向：悬疑反转
字数：8000 字
```

期望：

- 其他字段仍可为空。
- Agent 不要求继续补充参数。
- 完成后自动跳转编辑器。
- 正文符合所选方向和字数范围。
- 作品元数据保存用户选择的可选项。

### 13.4 设置验收

期望：

- Kimi 连接可用。
- DeepSeek 连接可用。
- API Key 不进入 Git。
- `.env.local` 被忽略。

### 13.5 极简体验验收

首页首屏不能出现：

- 数据看板。
- 风向标。
- 趋势列表。
- 过程卡片。
- 复杂参数。
- 作品统计。

用户从进入首页到开始生成，最多 2 次点击。

## 14. 迭代计划

### 14.1 V2 MVP

目标：先让两种模式能生成全文并跳转编辑器。

任务：

1. 重做首页为两模式极简入口。
2. 隐藏复杂导航。
3. 新增统一全文生成接口。
4. 重建生成内核，不沿用旧的固定流程输出。
5. 生成完成后保存作品。
6. 自动跳转编辑器。
7. 设置页保留 Kimi/DeepSeek。
8. 全自动模式增加非必选创作方向、字数和禁写方向。

### 14.2 V2.1

目标：改善真实可用性。

任务：

1. 增加 job + polling 进度。
2. 增加生成中止。
3. 增加失败恢复。
4. 增加弱段自动重写。
5. 优化长正文生成速度和稳定性。

### 14.3 V2.2

目标：市场导向增强。

任务：

1. 增加市场策略库。
2. 将用户手动录入的后台数据转成策略，不做数据看板。
3. 将复盘结果转为下一篇生成约束。
4. 增加作品表现手动录入入口，但不做图表。

## 15. 开发约束

- 不删除用户作品数据。
- 不提交 `.env`、`.env.local` 或任何真实 Key。
- 不绕过登录、验证码或平台限制。
- 不复制整篇受版权保护作品。
- 不提供规避审核、规避检测或低质量批量垃圾内容能力。
- 保持本地优先。
- 默认服务端口保持 Web 3000、API 3001。

## 16. 成功定义

V2 成功不以页面功能多少衡量，而以以下结果衡量：

- 用户几乎不用配置参数。
- 用户能一键生成完整短篇初稿。
- 生成后直接进入编辑器。
- 前端不干扰创作。
- Agent 生成结果更像面向平台读者的小说，而不是创作教学表格。
- 后续能根据作品表现和人工改稿持续调整生成策略。
